二、 會議紀錄內容
議案一：Headquarter.ai Autonomous Agent 平台與智慧審查規劃分享
【說明摘要】
Headquarter.ai 團隊針對其自主型多代理人平台進行專題演講。重點在於如何將 AI 從單純的「對話式機器人（Chatbot）」轉化為能夠處理複雜法規邏輯、具備執行力的「智慧代理人（Agent）」。
【重點論述】
多代理人架構（Multi-Agent Architecture）：
平台核心並非單一的大語言模型（LLM），而是由多個各司其職的 Agent 組成的工作流。
在醫療器材審查場景中，可分為「法規檢索 Agent」、「臨床數據分析 Agent」、「合規性交叉比對 Agent」與「最終報告撰寫 Agent」。
各 Agent 具備明確的責任邊界，透過相互協作（Orchestration）完成任務，大幅減少幻覺（Hallucination）風險。
資料主權與 BYOA 部署：
針對公部門機敏資料，平台支援「Bring-Your-Own-Account (BYOA)」模式。
所有運算流程皆運行在客戶（如 TFDA 或工研院）內部的 AWS Sub-account，資料不外流至第三方伺服器，確保符合資安規範。
視覺化工作流與 Agent Hub：
展示了「視覺化工作流編輯器」，非技術人員亦可調整 SOP 邏輯。
介紹了 Agent Hub 中的「稅法小書僮」成功案例，該案已實現 95% 以上的判讀準確率，並將作業時間從數小時壓縮至 5 分鐘。
AI 護欄與治理機制：
整合了 Amazon Bedrock Guardrails。當 AI 輸出涉及敏感個資或非預期的法規建議時，系統會自動攔截。
提供完整的稽核日誌（Audit Logs），每一項 AI 推理步驟均可追溯，滿足政府合規要求。
議案二：工研院採用 Autonomous Agent 服務與硬體資源規劃
【說明摘要】
工研院分享 116 年度「人工智慧應用於醫療器材查驗登記審查」之技術實作計畫。
【技術細節與決議】
混合雲架構與本地算力：
工研院將投入本地端 NVIDIA DGX1（配備 2x Tesla V100 32GB GPU）作為運算核心。
部署地端 LLM（如 Llama 3.2 90B 或 Gemma 4 31B），負責處理對隱私高度敏感的原始報批文件。
雲端部分由 AWS 進行帳號管理與權限控管，確保系統的高可用性。
推理引擎最佳化：
計畫採用 vLLM 作為推理後端，並透過量化技術（AWQ/GGUF）優化 Llama 3 等模型，以適應 V100 的 VRAM 限制。
預計達成推理速度 > 15 tokens/sec，支援多位審查員同時連線操作。
安全通訊協定：
採用 Cloudflare Tunnel 技術，將地端 GPU 算力安全地曝露給雲端前端，無需開放公網 IP，防範 DDoS 與非法入侵。
原型展示觀摩：
當日初步展示了「醫材審查 Agent」雛形，能夠從 510(k) 報批文件中自動識別設備分類代碼（Product Code）與適用法規標準。
議案三：醫療器材上市許可文件智慧申辦系統 (SaaS)
【說明摘要】
此議案旨在建立一個面向產業界（醫材商、顧問公司）的智慧申辦平台，將法規諮詢與預審流程數位化。
【核心功能規劃】
自動化申請書生成（Module 2）：
系統須從原始 CSV/JSON 數據中，精確提取 20 個關鍵法規實體（Entities），包含設備名稱、分類（I, II, III）、適應症、生物相容性標準等。
生成結構化的 Markdown 格式申請書，支持雙欄對照（Markdown 原始碼 vs. 渲染後視圖）。
智慧法規導引（Guidance Lab）：
支持 PDF 掃描檔的智慧 OCR（由 GPT-4o 或 Gemini Vision 驅動）。
獨家研發「Coral Keyword」功能，自動標記指引文件中的關鍵義務詞（如：MUST, SHALL, 應具備），輔助人工判讀。
Agentic Reviewer 預審引擎：
提供「Human-in-the-Loop」機制。顧問可選擇特定的 AI 模型（如 Claude 3.5 Sonnet 用於邏輯推論）對草稿進行自我稽核。
系統會產出「合規性矩陣（Compliance Matrix）」，自動指出草稿中缺失的實驗證明或不符指引的描述。
視覺化設計（Sleek Interface）：
採用「櫻花（Sakura）」主題配色與「Bento Grid（便當盒格柵佈局）」。
提供 Live Log 監控窗口，讓用戶能即時看到 Agent 的思考過程（Thinking Process），增加系統透明度與信任感。
議案四：國家藥物韌性整備計畫 (116 年度鉅額專案)
【說明摘要】
配合行政院重要政策，建立國家級「藥品/醫療器材韌性預警系統」，目標是打造類似 Palantir Foundry 的本體論（Ontology）決策環境。
【計畫執行深度細節】
背景與必要性：
因應地緣政治緊張及全球供應鏈高度不確定性，關鍵醫材（如葉克膜耗材、洗腎設備、防護物資）的自給自足至關重要。
計畫核心在於建立「平戰轉換」機制，確保緊急時期醫療防線不崩潰。
專案輔導法規團隊（Task Force）：
將成立專屬輔導團隊，包含 5 名以上駐點專業人員（碩士以上學歷，涵蓋生醫工程、資工、化學背景）。
設立專用辦公場域，負責滾動式檢討執行規範。
關鍵品項監測與研析：
擇定至少 3 項「特定關鍵醫療器材」進行優先輔導。
每季產出國際趨勢研析報告，盤點國內潛力業者名單（Seed List）。
技術驗證與臨床前測試：
計畫至少輔導 3 個案完成驗證。
啟動至少 15 件臨床前測試（Pre-clinical tests）及 2 件臨床評估，相關經費納入本專案支應。
AI 預警與決策行動方案：
利用 Headquarter.ai 平台建立「韌性預警 Agent」。
系統需整合海關數據、產能數據與國際庫存資訊，自動產出後續行動方案（Action Items），如「啟動第二供應商徵用」或「加速平行輸入審查」。
三、 深度討論與 Follow-up 問題彙整
在會議後半段，針對技術實作與政策落地提出以下 20 個關鍵問題與初步答覆：
Q：地端 Llama 3 推理速度若不如預期怎麼辦？
A：將導入 FP8 或 4-bit 量化，並優化 KV Cache 管理，必要時增加 DGX1 的算力節點。
Q：Cloudflare Tunnel 連線不穩會影響審查嗎？
A：系統前端將實作「斷線重連」與「狀態保存」機制，確保長文本處理不會因網路抖動而中斷。
Q：如何防止機敏文件流向雲端模型？
A：將建立「敏感資料路由機制」，僅將非去識別化、非機敏的 UI 控制請求發往雲端，核心內容鎖定在地端 GPU 處理。
Q：AI 是否會對醫療專業術語產生誤解？
A：計畫透過 LoRA 技術，針對 TFDA 歷年指引與 FDA 510(k) 數據對模型進行微調（Fine-tuning）。
Q：如何確保 Agent 的「Stop」機制有效？
A：後端 FastAPI 將結合 Threading 控制，當用戶點擊停止，立即發送訊號終止地端 GPU 推理進程。
Q：Bento Grid 佈局在小螢幕上會跑版嗎？
A：採用 RWD 響應式設計，針對 13 吋筆電與 27 吋螢幕進行斷點測試，確保 UI 體驗一致。
Q：多用戶併發時 DGX1 顯存如何管理？
A：設定 vLLM 的 max_num_seqs 參數，實施請求排隊機制，避免 Out-of-Memory (OOM) 崩潰。
Q：法規要求臨床數據絕對不能上雲，系統如何管控？
A：系統將設置硬性「隔離標籤」，特定類型的文件上傳後僅觸發本地運算邏輯。
Q：視覺化報告能下載成 PDF 嗎？
A：可以。將整合 ReportLab 或 Playwright 套件，將 Markdown 與 Plotly 圖表轉換為高品質 PDF。
Q：若 Gemini API 全球停機，系統有備援嗎？
A：有。系統具備「模型退避（Fallback）機制」，會自動切換至地端模型處理基礎任務。
Q：系統指令（System Instructions）如何更新？
A：SKILL.md 將版本化管理，存儲於本地資料庫，確保審查標準的一致性與可追溯性。
Q：是否支援 Redis 緩存以降低 GPU 負載？
A：是，針對重複性高的法規條文檢索，將實施快取機制。
Q：繁體中文翻譯在不同模型間有差異嗎？
A：地端 Llama 3 對繁中支持較弱，將透過 System Prompt 進行術語約束，或結合雲端模型進行語言修補。
Q：Live Log 是否會記錄 AI 的錯誤？
A：是的，日誌會存儲於 CloudWatch，用於後續分析 Agent 在哪個步驟發生邏輯偏移。
Q：櫻花主題能切換成傳統商務風格嗎？
A：提供「主題切換器（Jackpot）」，用戶可依喜好切換配色。
Q：未來增加第二台伺服器如何做負載平衡？
A：將在轉發層加入 Nginx 負載平衡器。
Q：長文件運算會導致瀏覽器逾時嗎？
A：採用異步（Asynchronous）處理模式，用戶可先行關閉視窗，運算完成後透過推播或郵件通知。
Q：前端動畫（Framer Motion）會影響效能嗎？
A：會嚴格控管 CSS 動畫資源消耗，確保在一般商用筆電流暢運行。
Q：是否有用戶反饋機制？
A：每個 Agent 輸出均附帶 👍/👎 按鈕，收集數據以優化 Prompt。
Q：如何適應新的 MCP 模型上下文協定？
A：架構設計保留高度彈性，接口層支援快速對接未來標準化 API。
